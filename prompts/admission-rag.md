# Промпт: помощник приёмной комиссии вуза (RAG + intent classifier)

**Стек:** FAISS + langchain + Cohere/OpenAI embeddings + Gemini 2.5 Flash (через OpenAI-совместимый эндпоинт `/v1beta/openai/`).
**Архитектура:** intent classifier → ветвление на «список специальностей» vs «общий вопрос».
**Источник:** admission-bot `zhezuRAG.py`

## Зачем

Telegram-бот приёмной комиссии вуза. Отвечает на вопросы абитуриентов **строго на основе официальных документов** (приказы, проспекты, положения), маршрутизирует «список специальностей» отдельным веткой, добавляет контактный footer.

## Системный промпт (general Q&A)

```
Ты помощник приемной комиссии {UNIVERSITY_NAME}. Отвечай точно и полно на основе предоставленных документов, представляя университет.
```

User-сообщение собирается из шаблона:

```
Вопрос абитуриента: {question}

КОНТЕКСТ ИЗ ДОКУМЕНТОВ:
{retrieved_chunks_with_sources}

Ответь на вопрос абитуриента. Если в документах нет ответа — честно скажи, что нужно уточнить в приёмной комиссии. Не выдумывай факты. В конце добавь контакты приёмной комиссии.
```

## Footer (приклеивается к каждому ответу)

```
📞 **За подробностями обращайтесь ко мне или в приемную комиссию:** {PHONE}
🌐 **Наш сайт:** {SITE}

💡 *Список составлен на основе {N} официальных документов нашего университета*
```

## Intent classifier (предварительная маршрутизация)

```
Классифицируй вопрос абитуриента в одну из категорий:
- SPECIALTIES_LIST — спрашивает список специальностей / направлений / факультетов
- GENERAL — общий вопрос (документы, сроки, экзамены, общежитие, оплата и т.д.)

Вопрос: "{question}"

Ответь одним словом: SPECIALTIES_LIST или GENERAL.
```

**Модель для классификации:** быстрая и дешёвая (`llama-3.1-8b-instant` через Groq, или `gemini-2.5-flash-lite`). Классификация → ветвление на узкоспециализированный промпт.

## Промпт для SPECIALTIES_LIST ветки

```
Составь полный список всех специальностей {UNIVERSITY_NAME} на основе официальных документов ниже.

ДОКУМЕНТЫ:
{specialty_docs}

Формат:
- Группируй по факультетам / институтам / уровням (бакалавриат / магистратура / докторантура)
- Для каждой специальности укажи код (если есть в документе) и полное название
- Не добавляй специальности, которых нет в документах
- В конце укажи общее число специальностей

Стиль: маркированный список, лаконично.
```

## Как переиспользовать

1. **Замена сегмента**: вместо университета — школа, медцентр, госорган, юрфирма. Логика та же: «отвечай по документам, не выдумывай, добавляй контакты».
2. **Intent classifier** легко расширяется: для школы — `ENROLLMENT_DATES` / `TUITION` / `SCHOOL_LIFE`; для клиники — `APPOINTMENTS` / `INSURANCE` / `DOCTORS`. Каждая ветка получает свой узкоспециализированный промпт.
3. **Источник документов**: PDF/DOCX/HTML — конвертировать в plain text + metadata (название документа, дата, тип). FAISS хранит embeddings; в metadata кладётся `source` для цитирования.
4. **Footer**: единый шаблон с эмодзи + телефоном + сайтом. Хорошо работает для **доверия** — пользователь видит «откуда» (сайт) и «куда дальше» (телефон).
5. **«Не выдумывай факты»** — ОБЯЗАТЕЛЬНАЯ строчка в промпте. Без неё RAG галлюцинирует, особенно по числам и срокам.
6. **Уверенность ответа** — если retrieved chunks пустые или короткие → честно сказать «нужно уточнить» + footer с телефоном. Не пытаться отвечать «из общих знаний модели».

## Связанные паттерны

- Если есть локальная зеркальная модель (`zhezuni v7.x` через Ollama) — fallback bridge: сначала локальная, при ошибке/таймауте — на облачную (Gemini/Groq). См. `admission-bot/zhezuni_backend.py` как пример.
- Для приватных вопросов (стоимость, личные данные студента) — вообще не отвечать через RAG, а сразу эскалировать на телефон/живого оператора.
