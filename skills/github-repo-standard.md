# Стандарт оформления GitHub-репозитория

Единый стандарт для хакатонных и продуктовых проектов (Solana / Colosseum и аналогичных). Все коммиты, PR, описания, issues — **на русском языке**.

---

## 1. Принцип

GitHub — это не склад файлов, а витрина проекта: **Product + Proof + Pitch**. Судья / инвестор / ревьюер должен за 60 секунд понять:

1. Что это за проект
2. Какую проблему решает
3. Как запустить
4. Как устроен
5. Почему это интересно

Если не отвечает — проект уже проигрывает.

---

## 2. Структура репозитория

```
project-name/
├── README.md
├── CONTRIBUTING.md      # сигнал open-source
├── LICENSE              # MIT / Apache-2.0
├── .env.example         # БЕЗ реальных секретов
├── .gitignore
├── .github/
│   └── workflows/       # CI/CD
├── docs/
│   ├── architecture.md
│   ├── product.md
│   ├── api.md
│   └── roadmap.md
├── assets/              # скриншоты, GIF, схемы, лого
├── programs/            # Solana / смарт-контракты
├── frontend/
├── backend/
└── tests/
```

**Правила:**
- Чёткая иерархия. Никаких `final_v2/`, `copy/`, `temp/`, файлов в корне без причины.
- Логичные имена. Только английский, kebab-case или snake_case.
- Всё на своём месте.

---

## 3. README — обязательная структура

README — лендинг проекта. Читают за 30 секунд.

````markdown
# Название проекта

[![Build](badge)] [![License](badge)] [![Solana](badge)]

> Однострочное описание: что это и зачем.

## Проблема
Что не так в текущем мире.

## Решение
Как проект это исправляет.

## Why Solana / Why <Stack>   ← ОБЯЗАТЕЛЬНЫЙ блок для Solana
Speed, cost, ecosystem, composability — конкретно, не общими словами.

## Demo
- Live: <ссылка>
- Видео: <ссылка>
- Скриншоты: ниже

![screenshot](assets/screenshot-1.png)

## Архитектура
Схема + краткое описание компонентов.

## Quick Start
Запуск за 3 команды:
```bash
git clone <repo>
cp .env.example .env
npm install && npm run dev
```

## Roadmap
- [x] MVP
- [ ] Фича X
- [ ] Mainnet

## Команда / Ссылки
Twitter, сайт, контакты.
````

**Бейджи в шапке — первое, что видит ревьюер.** Зелёный `build passing` весит больше абзаца текста.

---

## 4. Папка docs/

Минимальный комплект:
- `architecture.md` — как устроен продукт (схема + объяснение).
- `product.md` — что это и для кого.
- `api.md` — как с этим работать.
- `roadmap.md` — куда движется проект.

Даже простой `docs/` повышает воспринимаемую серьёзность проекта.

---

## 5. assets/

Люди мыслят визуально. Без визуала проект забывается.

Обязательно:
- Скриншоты UI
- Схема архитектуры
- Demo GIF (живой продукт в действии)
- Лого

**GitHub Pages** — бесплатное live demo прямо из репо.

## 6. Коммиты (на русском, Conventional Commits)

Плохо: `fix`, `update`, `aaa`, `final`, `wip2`
Хорошо:

```
feat: добавить модуль стейкинга
fix: исправить расчёт комиссии в swap
docs: обновить README — раздел Quick Start
refactor: вынести логику авторизации в отдельный сервис
test: добавить тесты для модуля платежей
chore: обновить зависимости
ci: настроить сборку через GitHub Actions
style: отформатировать код через prettier
perf: оптимизировать запросы к БД
```

**Правила:**
- Префикс на английском (стандарт), описание — на русском.
- Глагол в инфинитиве: «добавить», «исправить», «обновить».
- Без точки в конце.
- Длина заголовка ≤ 72 символа.
- Тело коммита (если нужно) — через пустую строку, объясняет «зачем», а не «что».

## 7. Pull Request — на русском

```markdown
## Что сделано
Краткое описание изменений.

## Зачем
Какую задачу / проблему решает.

## Как проверить
1. Шаг 1
2. Шаг 2

## Скриншоты / демо
(если есть UI-изменения)

## Чеклист
- [ ] Тесты проходят
- [ ] Документация обновлена
- [ ] .env.example актуален
```

## 8. Issues — на русском

**Bug:**
```
[BUG] Краткое описание

Шаги воспроизведения:
1. ...
2. ...

Ожидаемое поведение: ...
Фактическое поведение: ...
Окружение: OS, версия, браузер
```

**Feature:**
```
[FEATURE] Краткое описание

Проблема: что не так сейчас
Решение: предлагаемая реализация
Альтернативы: что рассматривали ещё
```

## 9. CI/CD — обязательно

Минимальный `.github/workflows/ci.yml`:

```yaml
name: CI

on: [push, pull_request]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci
      - run: npm run build
      - run: npm test
```

15 минут настройки = вид недель серьёзной разработки.

## 10. Сигналы «живого» репозитория

| Параметр | Плохо | Хорошо |
|----------|-------|--------|
| Коммиты | `fix`, `aaa` | `feat: добавить X` |
| Contributors | 1 человек | вся команда |
| Activity | всё залито за 1 день | равномерный график |
| Releases | нет | `v1.0.0` + changelog |
| Topics | нет | `solana`, `defi`, `web3`, `colosseum` |
| Badges | нет | build, license, stack |

---

## 11. Безопасность — критично

- **НИКОГДА** не коммитить реальные секреты, API-ключи, приватные ключи, мнемоники.
- `.env` всегда в `.gitignore`.
- В репозитории только `.env.example` с пустыми / фейковыми значениями.
- Если секрет утёк — немедленно ротировать ключи, недостаточно просто удалить коммит.

---

## 12. Releases и Topics (5 минут — большой эффект)

**Release:**
- Тег `v1.0.0` (semver).
- Changelog с разделами: Added / Changed / Fixed.

**Topics** (выставить в настройках репо):
- Технологии: `solana`, `rust`, `typescript`, `nextjs`
- Категория: `defi`, `nft`, `dao`
- Контекст: `hackathon`, `colosseum`, `web3`

## 13. Чеклист перед публикацией / сабмитом

**Структура:**
- [ ] README заполнен полностью
- [ ] Блок Why Solana / Why Stack есть
- [ ] Quick Start реально работает
- [ ] `.env.example` без реальных ключей
- [ ] `docs/` или architecture описаны
- [ ] LICENSE добавлен

**Сигналы:**
- [ ] Все члены команды — contributors
- [ ] GitHub Topics выставлены
- [ ] Release / Tag создан
- [ ] Badges добавлены
- [ ] CI проходит (хотя бы build)
- [ ] Demo-ссылка работает
- [ ] Скриншоты в README
- [ ] Описание (About) репо заполнено + ссылка на demo

**Безопасность:**
- [ ] В истории нет секретов (`git log -p | grep -i "key\|secret\|token"`)
- [ ] `.gitignore` корректный

---

## 14. Главное правило

Структуру репо делать **в первый час** проекта, не в последний. README-скелет + CI + папки → дальше просто наполнять.

Сильный GitHub не выигрывает сам по себе, но слабый GitHub убивает сильный проект.
